# 섹터 분류 출처 결정 기록

## 제목: 회사 마스터의 섹터 분류 데이터를 어디서 얻을지
- **날짜**: 2026-04-25
- **상태**: 검토 중 (결정 보류, 추가 조사 필요)

### 컨텍스트
- 새 디자인(`DART Live Styles.html`)이 **섹터 기반 가중치 점수 랭킹** 위젯을 핵심으로 둠 (중장기 투자자 페르소나 대응)
- mock 데이터에서 `sector: '반도체' | '2차전지' | '바이오' | '자동차' | '인터넷' | …` 같이 **사람이 읽기 쉬운 큰 묶음**으로 사용
- Company 도메인(E1) 스키마에 `sector` 컬럼 추가가 필요해짐
- 결정 대상: **이 섹터 값을 어떤 출처에서 받아올 것인가**

### 디자인이 가정하는 섹터 입자(粒度)
mock data 기준 약 **15~25개 큰 묶음**:
- 반도체, 반도체장비, 2차전지, 2차전지소재, 바이오, 자동차, 자동차부품, 인터넷, 게임, 철강, 화학, 상사, 전기전자, …

→ **GICS Sector(11개)는 너무 거침**, **한국표준산업분류 세분류(약 1,200개)는 너무 잘게**. 그 사이 어딘가가 필요.

### 후보 비교

| 출처 | 입자 | 자동화 | 비용 | 섹터 표현력 | 갱신 |
|---|---|---|---|---|---|
| **DART `induty_code` (한국표준산업분류)** | 너무 잘게 (1,200+) | API로 자동 | 0 | 약 (재매핑 필요) | 회사 변경 시 |
| **KRX 업종분류** | 적절 (~80개 KOSPI/~50개 KOSDAQ) | CSV 다운로드 (반자동) | 0 | 강 | 분기 |
| **GICS / FactSet 섹터** | 너무 거칠 (11/24/72) | 무료 X | 유료 | 약 (한국 산업 특수성 ✕) | — |
| **수동 매핑 테이블** | 자유 설정 | 수동 | 시간 | 최강 | 수동 |
| **하이브리드 (KRX 1차 → 수동 보정)** | 적절 + 자유 | 반자동 | 시간 일회성 | 강 | 분기 + 수동 |

#### 후보별 상세

**1. DART `induty_code` (한국표준산업분류 KSIC)**
- DART `company.json` 응답에 들어있음 (예: 삼성전자 `264` = 반도체 및 전자부품 제조업)
- 장점: 회사 마스터 시드(E1)와 같은 호출에서 바로 가져옴 → 추가 인프라 0
- 단점:
  - **1,200개 이상의 세분류** — 디자인의 ~20개 큰 묶음과 맞지 않음
  - 매핑 테이블을 우리가 직접 만들어야 함 (`264 → '반도체'`, `293 → '자동차부품'` …)
  - 한국 산업 트렌드 분류와 1:1 매칭 안 되는 케이스 많음 (예: 2차전지 소재 기업이 화학 분류로 들어가 있음)
- 결론: **단독으로는 부적합**. 매핑 노력이 KRX 분류 가져오는 것보다 큼.

**2. KRX 업종분류**
- KRX 정보데이터시스템(`data.krx.co.kr`)에서 CSV 다운로드 가능
- 한국 시장 관습에 맞춘 분류 (예: "전기전자", "운수창고업", "의료정밀")
- 장점:
  - 한국 시장 도메인 표준
  - 입자가 적당 (KOSPI 약 80개, KOSDAQ 약 50개)
  - **개미 투자자가 이미 익숙한 용어**
- 단점:
  - 공식 Open API 미존재 → 자동화 어려움 (CSV 수동 다운로드 또는 비공식 endpoint)
  - 다운로드 한 번에 전 종목 → 분기마다 갱신
  - 디자인의 *"2차전지 vs 2차전지소재"* 같은 세분 표현은 부분적으로만 가능 (KRX는 "전기전자"로 묶음)
- 결론: **베이스 분류로 가장 유력**. 다만 디자인 톤에 완전히 맞지는 않음.

**3. GICS (Global Industry Classification Standard)**
- 글로벌 표준 (S&P + MSCI), 11 sectors / 24 industry groups / 72 industries
- 장점: 글로벌 비교 시 표준
- 단점:
  - **유료 라이선스** (개인 프로젝트 부적합)
  - **한국 산업 특수성 표현 약함** (2차전지·반도체장비 같은 한국형 큰 섹터가 없거나 묻힘)
  - 데이터 수급 자체가 어려움
- 결론: **부적합**.

**4. 수동 매핑 테이블**
- 우리가 ~20개 섹터 직접 정의 → 각 종목 수동 매핑
- 장점:
  - 디자인과 완벽 일치 가능
  - 산업 트렌드(예: 2차전지 소재) 반영 자유
- 단점:
  - 2,500종목 수동 매핑 = 비현실적 시간
  - 신규 상장사 들어올 때마다 갱신 필요
- 결론: **단독으로는 부적합**. 보정용으로만.

**5. 하이브리드 — KRX 1차 + 수동 보정 + 매핑 테이블** ⭐ 유력
- KRX 업종을 베이스로 적재 → 우리가 정의한 ~20개 큰 묶음으로 매핑 → 특이 케이스 수동 보정
- 장점:
  - **자동화된 1차 분류** (KRX CSV로 80% 커버)
  - **디자인 톤에 맞는 큰 묶음** (매핑 테이블에서 결정)
  - **수동 보정 여지** (예: 에코프로비엠을 "화학"이 아니라 "2차전지소재"로)
- 단점:
  - 매핑 테이블 정의 일회성 노력 (~1일)
  - KRX 데이터 가져오는 자동화 까다로움 (CSV 수동 또는 비공식 API)

### 결정 (제안 — 추가 조사 필요)

**유력 방향: 하이브리드(5번)**

1. **저장 구조** (Phase 1 회사 마스터에 포함):
   ```
   company_version
     ...
     krx_sector       VARCHAR(40)   -- KRX 원본 (예: "전기전자")
     internal_sector  VARCHAR(40)   -- 우리 매핑 (예: "반도체")
   ```
   
   또는 별도 테이블:
   ```
   sector_mapping
     krx_code   PK
     krx_label  
     internal_sector
     manual_override  BOOLEAN
   ```

2. **수집 흐름**:
   - 시드 1회: KRX CSV 수동 다운로드 → resources/seed-data/krx-sector.csv
   - DART `corp_code`와 KRX `종목코드`로 조인
   - 매핑 테이블(자체 ~20개 internal_sector 정의)로 변환

3. **갱신**:
   - 분기 1회 KRX CSV 갱신 (수동)
   - 매핑 테이블은 거의 안 변함 (산업 트렌드 변경 시만)

### 결정 보류 사유

확인 필요한 항목:
1. **KRX CSV 자동화 가능성**
   - data.krx.co.kr 비공식 endpoint(POST OTP → 다운로드) 활용 — Java/Spring에서 구현 비용은?
   - pykrx 같은 Python 라이브러리 참조 가능
   - 자동화 어려우면 분기 수동 다운로드도 OK (개인 프로젝트 스코프)

2. **internal_sector 정의 — 몇 개로 할 것인가**
   - 디자인 mock에 등장하는 ~20개 + 추가 빈 카테고리 (기타) 1개?
   - 너무 많으면 가중치 점수 의미 희석, 너무 적으면 디자인 톤 안 맞음
   - 시드 데이터 분포 보고 결정

3. **DART `induty_code`도 같이 저장할 것인가**
   - 미래 분석에 유용할 수 있음 (산업 세분류 통계)
   - 저장 비용 거의 0이라 일단 같이 저장 권장

4. **수동 보정 빈도 추정**
   - KRX → internal 매핑이 잘 되는 종목 비중이 높으면 수동 보정 적음
   - 시드 후 분포 점검 단계 필요
